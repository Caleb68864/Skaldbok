## Critical Findings

### 1. Spec dropped 6 of 9 note types from design doc

**Root Cause:** spec_ambiguity
**Pattern:** new
**Destination:** spec amendment

The design doc explicitly says "All 9 note types in one table" with `typeData` JSON for type-specific fields (location, loot, rumor, quote, skill-check, recap). The forge phase reduced the Zod discriminated union to "at minimum: generic, npc, combat" (AC-S2-04). The "at minimum" qualifier allowed the spec to technically pass while silently dropping 6 types. The design doc places location/loot/rumor UI in Stage 3a, but the schema stubs should have been created in Stage 1 so the data model is production-correct on first write.

**Impact:** Schema will require a breaking evolution when these types are added. Any notes created before the schema expansion lack type-specific validation. Location capture during sessions — as common as NPC capture in a tabletop RPG — has no Quick Location button.
**Proposed Fix:** Amend AC-S2-04 to require all 9 discriminated union variants in the Zod schema: generic, npc, combat, location, loot, rumor, quote, skill-check, recap. Each variant's `typeData` shape should be defined (even if minimal) so the data model is forward-compatible. Add a "Quick Location" button to Notes hub (AC-S9) alongside Quick Note and Quick NPC.

### 2. Quick Notes body uses plain input, not Tiptap textarea

**Root Cause:** agent_limitation
**Pattern:** new
**Destination:** skill prompt improvement

AC-S9-02 specifies "Optional field: body (Tiptap editor)" for Quick Note creation. The implementation uses a plain `<input>` element rather than a `<textarea>` or the TiptapNoteEditor component (AC-S6-01). The core note body editing experience — rich text with @-mentions and wiki-links — is missing from the primary note creation flow.

**Impact:** Notes created via Quick Note have no rich text, no @-mentions, no wiki-links on export. The killer feature (Obsidian export with interlinked markdown) is broken for the most common capture flow. Quick NPC body field has the same issue.
**Proposed Fix:** Replace the plain `<input>` in QuickNoteDrawer and QuickNPCDrawer with either: (a) a `<textarea>` for lightweight plain-text capture, or (b) the TiptapNoteEditor component for full rich text. Given the design doc's emphasis on @-mentions and wiki-links, option (b) is correct. For worker skill prompts: when a spec says "body (Tiptap editor)", the worker must import and render the TiptapNoteEditor component — not use an HTML input element.

### 3. Pipeline abort at preflight with no recovery

**Destination:** factory config change

## Important Findings

### 4. Add Skills/Spells modals broken at runtime

**Root Cause:** agent_limitation
**Pattern:** new
**Destination:** new learning

The verify report shows AC-S11-05 (Heroic ability picker) and AC-S11-06 (Spell picker) as PASS based on structural/behavioral code analysis. However, the modals do not actually function at runtime — the data flow or rendering is broken when the user interacts with them.

**Impact:** Combat timeline's ability/spell pickers are non-functional, blocking the in-session combat journaling workflow.
**Proposed Fix:** Debug the AbilityPicker and SpellPicker components at runtime. Likely causes: (a) character data not flowing through CampaignContext to the picker, (b) activeParty.members not resolving linkedCharacterId to actual character records, or (c) the modal/drawer not receiving the correct props from CombatTimeline. The verify phase should include a "runtime smoke-test" acceptance criterion for all interactive components.

### 5. Combat conditions should be dropdowns from rulebook

**Root Cause:** spec_ambiguity
**Pattern:** new
**Destination:** spec amendment

AC-S11-03 specifies event types as "tap targets" including "condition" but does not specify that condition events should use a dropdown populated from Dragonbane's 6 conditions: Exhausted, Sickly, Dazed, Angry, Scared, Disheartened. The implementation likely renders a free-text input for condition events, violating the "no typing for quick-capture" design intent.

**Impact:** Condition logging requires typing instead of tapping, which is slow during combat and error-prone (misspellings won't match official condition names on export).
**Proposed Fix:** Amend AC-S11-03: when event type "condition" is selected, show a dropdown/chip-selector with the 6 Dragonbane conditions. The condition list should be sourced from the system definition or hardcoded for Dragonbane-only (per the design doc's "no multi-system abstractions" constraint).

### 7. No "all notes" campaign-wide export

**Root Cause:** spec_ambiguity
**Pattern:** new
**Destination:** spec amendment

The design doc Stage 2 export spec says: "single note, session, session+notes, all notes, clipboard copy." AC-S10 only covers single note (AC-S10-01), session (AC-S10-02), session+notes (AC-S10-03), and clipboard (AC-S10-04). Campaign-wide "all notes" export was dropped.

**Impact:** No way to bulk-export all notes from a campaign to Obsidian. User must export session-by-session, which is tedious for a multi-session campaign.
**Proposed Fix:** Add AC-S10-07: "Export All Notes" action on the campaign or Notes hub produces a .zip containing all notes from the active campaign, organized by session (subdirectories), plus a campaign index file (from renderCampaignIndex). Delivered via shareFile() with Blob download fallback.

### 8. NPC export/copy handlers are no-ops

**Root Cause:** agent_limitation
**Pattern:** new
**Destination:** new learning

The verify report itself flagged this as a SUGGESTION: "NPC notes in the Notes hub pass empty no-op functions `() => {}` for `onExport` and `onCopy` instead of the actual export/copy handlers." Export Note and Copy as Markdown are silently broken for NPC-type notes — the user clicks, nothing happens, no error shown.

**Impact:** NPC notes — one of the two primary note types — cannot be exported or copied to clipboard.
**Proposed Fix:** Wire `onExport` and `onCopy` props on NPC note cards in NotesScreen.tsx to the actual `exportNote()` and `copyNoteAsMarkdown()` handlers from `useExportActions()`. The verify phase should escalate "no-op handler" findings from SUGGESTION to IMPORTANT — a silently broken button is a functional bug, not a style issue.

### 11. No appears_in EntityLink for NPCs across sessions

**Root Cause:** spec_ambiguity
**Pattern:** new
**Destination:** spec amendment

AC-S4-06 only creates `contains` and `introduced_in` links. The design doc specifies that NPCs link to sessions via `appears_in` EntityLinks (separate from `introduced_in`). There's no mechanism to mark an NPC as appearing in a session they weren't introduced in.

**Proposed Fix:** Add a "Link NPC to session" action in the Notes hub or Session tab that creates an `appears_in` EntityLink between an NPC note and the active session. Update the export to build the NPC's full `sessions[]` list from both `introduced_in` and `appears_in` links.

### 12. Stale session warning is toast, not actionable modal

**Root Cause:** agent_limitation
**Pattern:** new
**Destination:** new learning

AC-S3-06 specifies the user should be "offered 'End it' or 'Continue' options" for stale sessions. The implementation shows a warning toast that auto-dismisses. The verify report marked this as PASS despite the criteria clearly requiring two action buttons.

**Proposed Fix:** Replace the stale session toast with a modal dialog containing two buttons: "End Session" (calls endSession()) and "Continue" (dismisses the modal). The verify phase should not mark criteria as PASS when the description specifies interactive options but the implementation provides a passive notification.
