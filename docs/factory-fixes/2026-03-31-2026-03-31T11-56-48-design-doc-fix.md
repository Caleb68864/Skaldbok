## Critical Findings

### 1. Path hallucination in spec generation — 5 of 6 preflight CRITs were wrong paths

**Root Cause:** agent_limitation
**Pattern:** new
**Destination:** skill prompt improvement

The forge agent scanned the codebase but produced wrong file paths in the spec: `features/character/` instead of `features/characters/` (plural), `features/notes/TiptapNoteEditor.tsx` instead of `components/notes/TiptapNoteEditor.tsx`, `features/character/CharacterSubNav.tsx` instead of `components/shell/CharacterSubNav.tsx`. The Skaldmark codebase uses both `features/` and `components/` directories — the agent defaulted to `features/` for everything. The prep agent copied these wrong paths into all 12 phase specs without verifying.

**Impact:** 5 phase specs had wrong paths; all would have failed at worker stage. Required ~10min human fix-up before Stage 4 could run.
**Proposed Fix:** (a) Stage 2 forge agent Step 3 (codebase analysis): after identifying files conceptually, verify every path by running `Glob` for the exact filename — confirm the path exists before writing it into the spec. (b) Stage 3 prep agent: verify every path in the `## Files` table exists before writing the phase spec. If a file doesn't exist, search for it by name and auto-correct. (c) Stage 3.5 preflight already catches these — no change needed there.

## Important Findings

### 3. Phantom sub-spec for already-implemented feature (SS-10)

**Root Cause:** spec_ambiguity
**Pattern:** new
**Destination:** skill prompt improvement

SS-10 specified replacing `===` with `startsWith()` in CharacterSubNav.tsx for active sub-nav detection. But line 27 of `src/components/shell/CharacterSubNav.tsx` already uses `location.pathname.startsWith(to + '/')`. The forge agent created a sub-spec for work that was already done. A worker could have introduced a regression by re-implementing existing logic differently.

**Impact:** One wasted worker spawn + verify cycle. Risk of regression on working code.
**Proposed Fix:** Add to Stage 2 forge agent codebase analysis: "Before creating a sub-spec for a behavior fix, read the target file and check whether the described behavior is already implemented. If the acceptance criteria are already satisfied by existing code, note it as 'Already Implemented — verification only' and either skip the sub-spec or convert it to a verify-only check."
