### 2. CombatTimeline interface contradiction makes SS-07 impossible

**Root Cause:** spec_ambiguity
**Pattern:** new
**Destination:** spec amendment

The spec simultaneously requires: (a) "keep CombatTimeline.tsx interface unchanged" (SS-07 constraint + escalation trigger) and (b) rewire CombatTimeline to read from encounter entities instead of note typeData. But `CombatTimeline.tsx` is self-contained -- it accepts `{ combatNoteId: string; onClose: () => void }` and internally calls `getNoteById(combatNoteId)` to load all data. There is no data-props interface to "keep unchanged" because the component doesn't accept participants, rounds, or events as props. The two constraints are architecturally contradictory.

**Impact:** SS-07 cannot be implemented as written. The escalation trigger would fire immediately, requiring human intervention regardless.
**Proposed Fix:** Amend the design doc and spec SS-07 with an architectural decision: either (a) refactor CombatTimeline to accept encounter data via props (requires escalation approval since it modifies the existing interface), or (b) build a new `CombatEncounterTimeline` component from scratch that reads from encounter entities, reusing lower-level UI primitives from CombatTimeline where possible. Option (b) is recommended -- it avoids the constraint violation, isolates risk, and allows parallel development.

## Important Findings

### 4. Bundle schema references 3 wrong import names/paths

**Root Cause:** agent_limitation
**Pattern:** recurring (wrong API/schema names in T11-56-48 fix #5, T18-21-40 context)
**Destination:** spec amendment

SS-02's `bundle.ts` has three compilation-blocking import errors:
1. `import { characterRecordSchema } from './character'` -- actual location: `schemas/character.schema.ts` (outside `src/types/`)
2. `import { partyMemberSchema } from './partyMember'` -- no such file; both `partySchema` and `partyMemberSchema` are in `src/types/party.ts`
3. `import { attachmentMetaSchema } from './attachment'` -- actual export name is `attachmentSchema`, and it includes a `blob: z.instanceof(Blob)` field that cannot survive JSON round-trip

**Impact:** SS-02 bundle.ts would not compile. All downstream import/export sub-specs (SS-09 through SS-15) depend on this file.
**Proposed Fix:** Amend SS-02: (a) import `characterRecordSchema` from `../../schemas/character.schema`, (b) import `partyMemberSchema` from `./party`, (c) import `attachmentSchema` from `./attachment` and create an `attachmentBundleSchema` using `.omit({ blob: true }).extend({ data: z.string().optional(), encoding: z.literal('base64').optional() })` for the serialized form.

### 5. Dexie migration guard uses wrong table name and access pattern

**Root Cause:** agent_limitation
**Pattern:** new
**Destination:** spec amendment

SS-01 migration code uses `await tx.table('meta').get('migration_v6_combat')`. Two errors:
1. Table is named `metadata`, not `meta`
2. `.get('migration_v6_combat')` looks up by primary key `id`, not the `key` field. The `metadata` table has schema `'id, &key'` with `MetadataRecord { id, key, value }`.

Correct pattern: `await tx.table('metadata').where('key').equals('migration_v6_combat').first()`

For the put operation: `await tx.table('metadata').put({ id: generateId(), key: 'migration_v6_combat', value: 'true' })`

**Impact:** Migration guard would either throw (wrong table name) or return undefined every time (wrong lookup), causing the migration to re-run on every app launch, duplicating encounter and creature template records.
**Proposed Fix:** Amend SS-01: replace all `tx.table('meta')` with `tx.table('metadata')`, replace `.get('key')` with `.where('key').equals('key').first()`, and use `generateId()` for the put operation's `id` field.

### 6. NPC note typeData has no stats fields -- migration produces empty creature templates

**Root Cause:** spec_ambiguity
**Pattern:** new
**Destination:** spec amendment

SS-01 NPC migration reads `note.typeData?.hp`, `note.typeData?.armor`, `note.typeData?.movement`, `note.typeData?.attacks`, etc. But `npcTypeDataSchema` in `src/types/noteValidators.ts` only defines `{ role?: string, affiliation?: string }`. There are no HP, armor, movement, attacks, abilities, or skills fields. The migration would produce creature templates with `stats: { hp: 0, armor: 0, movement: 0 }` and empty arrays for attacks/abilities/skills.

**Impact:** All migrated NPC creature templates would have zeroed-out stat blocks, defeating the purpose of migrating NPC data into the bestiary.
**Proposed Fix:** Amend SS-01: acknowledge that NPC notes in the current schema only contain `role` and `affiliation` in typeData. Migration should map: `name` from note title, `description` from note body (Tiptap JSON), `role` from `typeData.role`, `affiliation` from `typeData.affiliation`, `category: 'npc'`. Set `stats` to `{ hp: 0, armor: 0, movement: 0 }`, `attacks: []`, `abilities: []`, `skills: []` with a comment indicating these were not present in the original NPC note format and should be filled in by the user via the Bestiary UI.
